### Общие требования

Система представляет собой HTTP REST API со следующими требованиями к бизнес-логике:

* регистрация, аутентификация и авторизация пользователей;
* _**система групп и прав в ней (например member, admin, owner);**_
* возможность заходить в группы по ссылке;
* создание групп и возможность создания задач для группы/пользователя;
* создание записей о задаче и ее статусе (например todo, in work, done);
* просмотр изменений в задаче (кто поставил, кто последний изменил);

## Функционал проекта
### Минимум для диплома
- создание, регистрация пользователя;
- создание групп для авторизованных пользователей;
- создание и изменение задач для группы и пользователя;
- получение задач в группе;
- создание ссылок-приглашений для пользователей;
### Дополнительный функционал
- получение собственных групп;
- получение задач во всех группах к которым относится пользователь
- система прав в группе


## Абстрактная схема взаимодействия с системой

Ниже представлена абстрактная бизнес-логика взаимодействия пользователя с системой:

1. Пользователь регистрируется в системе.
2. Пользователь создает группу.
3. Пользователь создает приглашение.


### Сводное HTTP API

Task-система «godo» должна предоставлять следующие HTTP-хендлеры:

* `POST /api/user/register` — регистрация пользователя;
* `POST /api/user/login` — аутентификация пользователя;
* `POST /api/group/create` - создание группы пользователей;
* `GET /api/group/<slug>` - получение информации о группе;
* `POST /api/tasks/group/<slug>` - создание задачи для группы (или путь /api/group/<slug>/tasks);
* `GET /api/tasks/group/<slug>` - получение задач для группы (или путь /api/group/<slug>/tasks);
* `POST /api/user/tasks` - создание задач для пользователя;
* `GET /api/user/tasks` - получение задач для пользователя;
* `PATCH /api/tasks/<id>` - изменение задачи;
#### Дополнительные хендлеры
* `GET /api/users` - получение всех пользователей;
* `GET /api/user/<username/id>` - получение конкретного пользователя;
* `PATCH /api/group/<slug>` - изменение группы;
* `PATCH /api/group/<slug>/members/<id>` - изменение прав пользователя в группе; ---------
* `GET /api/group/members` - получение всех участников группы; --------------------------
* `DELETE /api/group/<slug>/members/<id>` - удаление пользователя из группы; -------------
* `POST /api/invites/use/<invite_id>` - использовать приглашение.

# Эндпоинты
#### **Регистрация пользователя**

Хендлер: `POST /api/user/register`.

Регистрация производится по паре логин/пароль. Каждый логин должен быть уникальным.
После успешной регистрации должна происходить автоматическая аутентификация пользователя.

Формат запроса:

```json
{
  "login": "<login>",
  "password": "<password>"
}
```

Возможные коды ответа:

- `200` — пользователь успешно зарегистрирован и аутентифицирован;
- `400` — неверный формат запроса;
- `409` — логин уже занят;
- `500` — внутренняя ошибка сервера.

#### **Авторизация пользователя**

Хендлер: `POST /api/user/login`.

Авторизация производится по паре логин/пароль. При успешной авторизации пользователю выдается или авторизационная
кука или bearer-token (пока что не решил что буду использовать).

Формат запроса:

```json
{
  "login": "<login>",
  "password": "<password>"
}
```

Возможные коды ответа:

- `200` — пользователь успешно аутентифицирован;
- `301` — неверная пара логин-пароль;
- `400` — неверный формат запроса;
- `500` — внутренняя ошибка сервера.

#### **Получение пользователя**

Хендлер: `GET /api/user/<username/id>`.

Возможные коды ответа:

- `200` — пользователь успешно найден:
```json
{
  "id": "<id>",
  "username": "<username>"
}
```
- `301` — неверная пара логин-пароль;
- `400` — неверный формат запроса;
- `500` — внутренняя ошибка сервера.

#### **Получение пользователей**

Хендлер: `GET /api/users`.

Возможные коды ответа:

- `200` — успешно выданы все пользователи:
```json
[
  {
    "id": "<id>",
    "username": "<username>"
  }
]
```
- `204` - пользователей
- `301` — неверная пара логин-пароль;
- `400` — неверный формат запроса;
- `500` — внутренняя ошибка сервера.

#### **Создание группы**
Хендлер: `POST /api/group/create`.

Создание группы доступно только авторизованным пользователям. У группы обязательно должны быть уникальные поля
name и slug.

Формат запроса:

```json
{
  "name": "<name>",
  "description": "<description>",
  "slug": "<slug>"
}
```
В запросе обязательным полем является только name, остальные поля по выбору.

Возможные коды ответа:

- `200` — группа успешно создана;
- `301` — пользователь не авторизован;
- `400` — неверный формат запроса;
- `500` — внутренняя ошибка сервера.

#### **Изменение группы**
Хендлер: `PATCH /api/group/<slug>`.

* 1 вариант. Изменение группы доступно только пользователям, с правом на изменение (дополнительный функционал,
добавлять его необязательно. Достаточно быть членом группы для изменения.).
* 2 вариант - группу может изменять только создавший ее пользователь.

Формат запроса:

```json
{
  "name": "<name>",
  "description": "<description>",
  "slug": "<slug>"
}
```
В запросе обязательно должно быть хотя-бы одно поле для изменения.

Возможные коды ответа:

- `200` — группа успешно создана;
  В ответе возвращается объект группы с измененными полями:
```json
{
  "name": "<name>",
  "description": "<description>",
  "slug": "<slug>"
}
```
- `400` — неверный формат запроса;
- `401` — пользователь не авторизован;
- `403` — пользователь не имеет прав на изменение;
- `404` — группа не найдена;
- `500` — внутренняя ошибка сервера.

#### **Получение информации о группе**
Хендлер: `GET /api/group/<slug>`. Вместо slug, для поиска можно использовать id.

Получение группы доступно только авторизованным пользователям[, которые состоят в группе].

Формат запроса:


В запросе обязательным полем является только name, остальные поля по выбору.

Возможные коды ответа:

- `200` — группа успешно создана;
Формат ответа:
```json
{
  "id": "<id>",
  "name": "<name>",
  "description": "<description>",
  "slug": "<slug>",
  "members": [
    {
      "id": "<id пользователя>",
      "username": "<имя пользователя>",
      "since": "<как давно пользователь находится в группе>",
      "is_admin": "<является ли пользователь администратором группы>"
    }
  ]
}
```
- `401` — пользователь не авторизован;
- `403` — пользователь не имеет прав на чтение;
- `404` — группа не найдена;
- `500` — внутренняя ошибка сервера.

#### **Создание ссылки-приглашения в группу(усложненный вариант)**
Хендлер: `POST /api/group/<slug>/invite`. _**Вместо slug, для поиска можно использовать id.**_

Создание приглашения в группу доступно только авторизованным пользователям, состоящим в группе.

Формат запроса:
json объект с выборочными полями.
Поля:
- limit [int] - количество людей, которые можно пригласить по ссылке(если передано 0, то лимита нет)
- expires [bool] - истекает ли ссылка через 14 дней. Если false, то не истекает _[еще можно сделать создание с полем expires_at]_.
 Пустое тело запроса. В ответе будет одноразовая ссылка-приглашение, при переходе по которой
авторизованный пользователь попадет в группу.
Возможные коды ответа:

- `200` — приглашение успешно создано;
  Формат ответа:
    Вернется ссылка-приглашение в формате json:
```json
{
  "invite": "<invite_link>"
}
```
- `401` — пользователь не авторизован;
- `403` — пользователь не имеет прав на приглашение новых участников;
- `404` — группа или пользователь не найдены;
- `500` — внутренняя ошибка сервера.

#### **Приглашение в группу**
Хендлер: `POST /api/group/<slug>/invite`. Вместо slug, для поиска можно использовать id.

Создание группы доступно только авторизованным пользователям.

Формат запроса:
JSON объект с одним обязательным полем по выбору. Если отправить пустое тело запроса, то в ответе
handler вернет ссылку-приглашение.
```json
{
  "user_id": "<user_id>",
  "username": "<username>"
}
```

Возможные коды ответа:

- `200` — пользователь успешно добавлен в группу;
  Формат ответа:
  Если не было создано приглашение конкретному пользователю, то вернется ссылка в формате json:
```json
{
  "invite": "<invite_link>"
}
```
- `401` — пользователь не авторизован;
- `403` — пользователь не имеет прав на приглашение новых участников;
- `404` — группа или пользователь не найдены;
- `500` — внутренняя ошибка сервера.


#### **Создание задачи для группы**
Хендлер: `POST /api/tasks/group/<slug>`

Создание задачи доступно только авторизованным пользователям, состоящим в группе 
(имеющим права на создание задач). Формат даты - RFC3339

Формат запроса:
```json
{
  "name": "<название задачи>",
  "description": "<описание задачи>",
  "exceed": "<крайний срок сдачи задачи>"
}
```
Поля запроса

Возможные коды ответа:

- `200` — задача успешно создана для группы;
- `401` — пользователь не авторизован;
- `403` — пользователь не имеет прав на создание задач;
- `404` — группа не найдена;
- `500` — внутренняя ошибка сервера.

#### **Получение задач в группе**
Хендлер: `GET /api/tasks/group/<slug>`

Создание задачи доступно только авторизованным пользователям, состоящим в группе
(имеющим права на создание задач). Формат даты - RFC3339.
В дальнейшем планируется добавить функционал фильтрации и сортировки в аргументах пути.
Минимум фильтрации - фильтр по статусу задачи вне зависимости от регистра (/api/tasks/group/<slug>?status=NEW)

Возможные коды ответа:

- `200` — задача успешно создана для группы;
Формат ответа:
```json
[
  {
    "id": "<id задачи>",
    "name": "<название задачи>",
    "description": "<описание задачи>",
    "exceed": "<оптимальный крайний срок сдачи задачи>",
    "created_by": "<id пользователя, который создал задачу>",
    "created_at": "<время создания>",
    "processor": "<id пользователя, который взял задачу(если такой есть)>",
    "status": "<статус задачи, один из предложенных>"
  }
]
```
Оптимальная сортировка выдачи - первичная сортировка по возрастанию по дате сдачи,
далее по возрастанию даты создания.
Возможные статусы:
* NEW - задача свободна и ее можно взять.
* PROCESSING - задачу уже взяли.
* DONE - задача уже выполнена.
- `401` — пользователь не авторизован;
- `403` — пользователь не имеет прав на создание задач;
- `404` — группа не найдена;
- `500` — внутренняя ошибка сервера.

#### **Изменение задачи**
Хендлер: `PATCH /api/tasks/<id>`

Изменение задачи доступно только авторизованным пользователям, состоящим в группе, к которой относится задача
(имеющим права на создание задач). Формат даты - RFC3339.
В дальнейшем планируется добавить функционал фильтрации и сортировки в аргументах пути.
Минимум фильтрации - фильтр по статусу задачи вне зависимости от регистра (/api/tasks/group/<slug>?status=NEW)

Возможные статусы:
* NEW - задача свободна и ее можно взять.
* PROCESSING - задачу уже взяли.
* DONE - задача уже выполнена.

Формат запроса:
```json
{
  "name": "<название задачи(изменить может только админ группы если таковые имеются или создатель задачи)>",
  "description": "<описание задачи(изменить может только админ группы если таковые имеются или создатель задачи)>",
  "status": "<статус задачи, один из предложенных[NEW,PROCESSING,DONE]>"
}
```

Возможные коды ответа:

- `200` — задача успешно изменена;
- `401` — пользователь не авторизован;
- `403` — пользователь не имеет прав на изменение задачи;
- `404` — группа не найдена;
- `500` — внутренняя ошибка сервера.

Возможные коды ответа:

- `200` — приглашение успешно получены приглашения;
  Формат ответа:

```json
[
  {
    "id": "<invite id>",

```
- `401` — пользователь не авторизован;
- `403` — пользователь не имеет прав на приглашение новых участников;
- `404` — группа или пользователь не найдены;
- `500` — внутренняя ошибка сервера.

Хендлер: `POST /api/user/invites/use/<uuid>`

Использовать приглашение могут только авторизованные пользователи, не состоящим в группе, к которой относится приглашение.

- `404` — приглашение не найдено;
- `500` — внутренняя ошибка сервера.

Использовать приглашение могут только авторизованные пользователи, не состоящим в группе, к которой относится приглашение.

Возможные коды ответа:

- `200` — пользователь успешно использовал приглашение;
- `204` - пользователь уже находится в группе;
- `401` — пользователь не авторизован;
- `404` — приглашение не найдено;
- `500` — внутренняя ошибка сервера.
